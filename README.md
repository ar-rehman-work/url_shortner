# URL Shortener API

## Overview

This project is a backend URL shortener service built using Ruby on Rails.
It is designed to demonstrate clean API design, deterministic short code generation, and system design trade offs in a simple but scalable architecture.

The system focuses on correctness, predictability, and performance while keeping the frontend completely decoupled.

This project is intentionally designed to explore core system design concepts such as idempotency, deterministic encoding, and scalable URL resolution.

## Problem Statement

The system solves the problem of converting long URLs into short, unique identifiers that:

- Are easy to store and retrieve
- Avoid collisions
- Provide fast redirection
- Maintain deterministic behavior

## Features

- Convert long URLs into short codes
- Deterministic short code generation using Base62 encoding
- Idempotent behavior (same long URL → same short code)
- Fast redirection using indexed lookup
- RESTful API design
- Simple and scalable schema

## Tech Stack

- Ruby on Rails (API-only backend)
- React.js
- PostgreSQL
- Base62 encoding for ID transformation
- RESTful API architecture
- JWT authentication

## Request Flow & API Endpoints

### Create Short URL

```text
Client → Controller → Model → Database → Response (short_code)
```

```http
POST /shorten
Authorization: Bearer <token>
```

#### Request

```json
{
  "long_url": "https://example.com"
}
```

#### Response

```json
{
  "short_code": "abc123"
}
```

---

### Redirect to Original URL

```text
Client → Controller → DB lookup → Redirect → Original URL
```

```http
GET /:shortcode
```

Redirects user to the original long URL.

## Architecture & Trade-offs

### 1. Short Code Generation Strategy

#### Base62 Encoding (ID based approach)

Short codes are generated by encoding the database ID using Base62.

**Why this approach:**

* Deterministic output
* No collisions
* No retry loops
* O(1) generation
* Simple mapping from ID → short code

**Pros:**

* Very fast
* Guaranteed uniqueness
* Easy to scale
* No randomness required

**Cons:**

* Predictable URLs (enumeration possible)
* Slight security limitation

---

#### Alternative: Random String Generation

**Pros:**

* Hard to guess URLs
* Good for non-sequential systems

**Cons:**

* Collision handling required
* Retry loops under load
* Non-deterministic behavior
* Extra database checks needed

---

#### Alternative: Hash-Based Approach

**Pros:**

* Deterministic output
* Same input always produces same hash

**Cons:**

* Long strings
* Needs truncation (collision risk)
* Less readable

---

### Final Decision

Base62 encoding was chosen because it provides the best balance of simplicity, performance, and deterministic behavior.

---

### 2. Idempotent URL Mapping

Same long URL always returns the same short code.

```ruby
  ShortUrl.find_or_create_by(long_url: url)
```

**Pros:**

* Prevents duplicate records
* Saves storage
* Consistent API behavior

**Cons:**

* Harder to support per-user variations
* Limits future expiry flexibility

### 3. Database Design

Single table design:

* id
* user_id
* long_url
* short_code
* timestamps

**Pros:**

* Simple schema
* Fast indexed lookups
* Easy to extend

**Cons:**

* Needs migration for analytics/expiry later

### 4. Authentication Design

JWT-based authentication is used for stateless API access.

**Flow:**

* User logs in
* Server returns JWT token
* Frontend stores token
* Token sent in Authorization header

**Pros:**

* Stateless backend
* Scalable
* Secure API access control

**Cons:**

* Token management required on frontend
* Logout is client-side invalidation
